# 中文 HTML 晨报合同

## 交付形式

任何被称为“晨报”或 “briefing” 的交付都生成 UTF-8、可独立打开、可浏览的完整 HTML。正常环境保存为本地 HTML 文件；环境或测试明确禁止写文件时，在回复中返回从 `<!doctype html>` 到 `</html>` 的完整 HTML 代码块，并紧邻说明“未保存文件”。禁止改成 Markdown 草案。

页面正文使用中文；证券代码、公司名、验证状态和英文来源名保留原文。所有证据使用可点击的原始页面链接，不链接搜索结果页或聚合摘要。

## 固定顺序

HTML 正文严格按以下顺序组织：

1. **一分钟结论**：今天是否需要行动、最重要的组合风险和“维持原计划”结论；最多 180 个中文字符。
2. **数据与覆盖**：最新成功导入时间、持仓截至时间、数据年龄、新鲜度状态、已扫描／总持仓、遗漏项和报告生成时间。
3. **重大市场事件**：最多 5 条。
4. **科技重大事件**：最多 4 条。
5. **重点持仓建议**：最多 6 个持仓。
6. **无重大变化**：汇总其余已扫描持仓，不逐项扩写；遗漏项必须另列原因。
7. **事件日历**：即将到来的持仓财报、宏观公布或已知关键日期，注明时区和日期来源。
8. **证据与风险**：证据清单、冲突／更正、数据限制、组合风险和研究辅助声明。

八个章节在短晨报、测试场景、数据不足和零来源时仍全部保留。缺值使用“未提供”“未验证”或“不适用”；空章节使用一句“无已验证项目”。不得通过省略章节或字段节省 Token。

## 硬性篇幅上限

- 一分钟结论：最多 180 个中文字符。
- 每条市场或科技事件的分析正文：最多 120 个中文字符。
- 每个重点持仓的分析正文：最多 180 个中文字符，不含结构化动作字段。
- 可见主分析正文合计：最多 2,200 个中文字符；标题、表头、URL 和证据审计表不计入该上限。
- 重大市场事件、科技事件、重点持仓分别硬限制为 5、4、6；不得通过拆表、附录或同义分类绕过。

超出上限的低优先级内容合并为“无重大变化”。证据审计表只保留正文实际引用的证据，不堆积候选链接。

## 事件卡片字段

每条市场或科技事件必须展示：

- 事件与发生时间；
- 验证状态；
- 来源名、`canonical_domain` 和原始链接；
- 事实摘要；
- 与组合的主要关联类别；
- 存在冲突时的差异和降级结果。

`primary_only_pending` 明确显示“等待独立确认”，`unverified_single_source` 明确显示“仅观察”，两者不得出现在动作依据栏。

没有任何候选 URL 或来源时，事件卡片显示 `evidence_gap: true`，验证状态显示“不适用”，来源与链接显示“未提供”。`evidence_gap` 是证据缺口标记，不是四种验证状态之一，不得把零来源标成 `unverified_single_source`，也不得让该事件进入动作依据。

## 持仓建议字段

每个重点持仓必须展示：代码、当前数量、当前市值、组合权重、影响类别、建议动作、触发条件、反证条件、时间范围、主要风险、证据状态、证据链接和判断理由。短晨报或字段缺失时仍展示全部字段，并分别填写“未提供”“未验证”或“不适用”，不得合并或省略。若为集中度再平衡，标记“组合风险动作，非当日新闻触发”。

## 覆盖、证据与新鲜度

- 页面顶部放置新鲜度徽标：`fresh`、`stale` 或 `unknown`，并紧邻展示成功导入时间和持仓截至时间。
- 只有存在最新成功 IBKR 导入时才展示正式 `scanned_holdings / total_holdings`；只有两者相等且遗漏为零时才可写“全持仓已覆盖”。
- 缺少最新成功导入时，把题设或用户列出的证券单列为“用户提供的临时范围”，正式覆盖显示“未验证”，不得生成正式覆盖分数或声称“全持仓覆盖”。
- 每个重要结论用稳定标识关联证据审计表。审计表包含来源、规范域名、原始链接、发布时间、抓取时间、验证状态及冲突说明。
- 数据为 `stale` 或 `unknown` 时，在页面顶部显示醒目警告，并将重点持仓动作统一限制为“观察／刷新数据后复核”。
- 没有足够证据时显示证据缺口，不用推测填满版面；零来源按 `evidence_gap` 规则处理。

## HTML 最低质量

使用语义化标题、表格或卡片，保证桌面和移动端可读。为数据新鲜度、验证状态和建议动作提供文字标签，不只依赖颜色。转义所有外部文本，禁止注入未经清理的 HTML。页面底部注明：“本报告仅供研究辅助，不构成收益保证或自动交易指令；任何交易由用户确认并在 IBKR 自行执行。”
